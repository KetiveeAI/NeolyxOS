🧠 Userspace Memory Management

Last Updated: January 1, 2026

This document explains how NeolyxOS handles memory for userspace processes (Ring 3), specifically regarding ELF loading, BSS sections, and the heap.

Virtual Address Space Layout

NeolyxOS currently uses an identity-mapped layout for simplicity during the early stages, but with specific regions for userspace separation.

Virtual Address Size Description Access
0x000000 - 0x400000 4MB Kernel / Low Memory Ring 0 Only
0x400000 - 0x600000 2MB Userspace Code & Data Ring 3 (User)
0x600000 - 0x680000 512KB Userspace Stack Ring 3 (User)
0xC0000000 Varies Framebuffer (Mapped Direct) Ring 3 (User)

Why 2MB for Code/Data?

Originally, only 1MB (0x400000 - 0x500000) was allocated. However, the Desktop environment (Neolyx Desktop) ran into a critical issue:

  1. ELF File Size: ~50KB (stored on disk).
  2. BSS Section: Large static arrays and uninitialized data.
    • 512KB Static Heap (heap[512*1024])
    • 192KB Window Thumbnails (wm_minimized_t[16])
  3. Total Memory Required: ~1.8MB

When the kernel loaded the ELF, the BSS section extended beyond 0x500000, causing a page fault or silent corruption when accessing variables like heap or g_window_states.

Fix: The kernel now maps 0x400000 to 0x600000 (2MB), covering the entire 1.8MB requirement.

📚 ELF Loading Process

The loader (src/core/elf.c) performs the following:

  1. Parse Header: Reads 64-bit ELF header.
  2. Load Segments: Copies LOAD segments from the embedded file data to memory at p_vaddr.
  3. Clear BSS: Zeroes out memory from p_vaddr + p_filesz to p_vaddr + p_memsz. This is where the 1.8MB expansion happens.

🧱 Heap Management

Currently, userspace applications use a static heap defined in nxsyscall.h:

// desktop/include/nxsyscall.h
static uint8_t heap[512 * 1024]; // 512KB Static Array

Limitations

  • Fixed Size: Cannot grow beyond 512KB.
  • BSS Bloat: Increases the process binary's memory footprint at load time (as seen above).

Future Roadmap: Dynamic Heap

To allow dynamic resizing:

  1. Implement sys_brk syscall in Kernel.
  2. Kernel tracks program_break (end of data segment).
  3. malloc calls sbrk to request more pages from OS.
  4. Kernel maps new pages on demand.

🥞 Stack Management

The stack is located at 0x600000 and grows downwards.

  • Top: 0x680000
  • Bottom: 0x600000
  • Size: 512KB

This is sufficient for deep recursion and large local variables in the current desktop shell.

⚠️ Common Pitfalls

  1. Floating Point (SSE): Userspace must be compiled with SSE enabled (-mno-red-zone only), otherwise float operations cause #UD (Invalid Opcode) exceptions because libc is missing and GCC defaults to SSE instructions.
  2. Freestanding: No libc exists. Functions like memcpy, memset, printf must be implemented locally or in a shared userspace library.
